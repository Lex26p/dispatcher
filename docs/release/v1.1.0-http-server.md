# v1.1.0-http-server

## Status

Released.

## Scope

This release adds the first real HTTP server layer for the dispatcher backend.

The implementation provides:

- `dispatcher-http` library
- Boost.Beast / Boost.Asio based HTTP server foundation
- HTTP request mapping to `dispatcher::api::TransportRequest`
- HTTP response mapping from `dispatcher::api::TransportResponse`
- transport-router based HTTP endpoint registration
- real TCP server application `dispatcher-http-server`
- manual smoke-testable HTTP endpoints

## Added application

### `backend/apps/dispatcher-http-server`

The new executable starts a Boost.Beast HTTP server.

Default manual run:

```powershell
.\out\build\windows-vs-debug\backend\apps\dispatcher-http-server\Debug\dispatcher-http-server.exe 127.0.0.1 18080

Available endpoints:

GET /health
GET /ready
GET /api/v1/runtime
GET /api/v1/alarms
HTTP endpoints
GET /health

Returns aggregate runtime health JSON.

Example shape:

{
  "status": "healthy",
  "ready": true,
  "check_count": 2,
  "healthy_count": 2,
  "degraded_count": 0,
  "unhealthy_count": 0,
  "invalid_count": 0
}
GET /ready

Returns readiness JSON.

Example shape:

{
  "status": "ready",
  "ready": true,
  "readiness_blockers": false,
  "invalid_checks": false,
  "check_count": 2
}
GET /api/v1/runtime

Returns the current HTTP runtime endpoint skeleton response.

Example shape:

{
  "status": "available",
  "endpoint": "runtime",
  "path": "/api/v1/runtime",
  "method": "GET",
  "source": "dispatcher-http"
}
GET /api/v1/alarms

Returns the current HTTP alarm endpoint skeleton response.

Example shape:

{
  "status": "available",
  "endpoint": "alarms",
  "path": "/api/v1/alarms",
  "method": "GET",
  "source": "dispatcher-http",
  "items": []
}
